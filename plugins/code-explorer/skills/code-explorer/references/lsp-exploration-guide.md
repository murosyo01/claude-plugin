# LSP 探索ガイド

コード探索で使う LSP 操作の詳細リファレンス。各操作の目的・引数・戻り値と、コードベース分析における具体的な活用パターンを記載する。

## LSP 操作一覧

### documentSymbols — ファイル構造の把握

ファイル内に定義されているすべてのシンボル（関数・クラス・変数・インターフェースなど）を階層的に返す。

**使いどころ:**
- 未知のファイルを最初に読む前に全体像を掴む
- エントリポイントとなる公開シンボルを素早く特定する
- ファイルの責務（何を定義・エクスポートしているか）を把握する

**使い方の流れ:**
1. `Glob` でファイルを特定する
2. `LSP documentSymbols` でシンボル一覧を取得する
3. 重要なシンボルに `Read` または `LSP hover` で詳細を確認する

**注意点:**
- 返されるシンボルはファイルスコープのもののみ。ローカル変数や関数内の定義は含まれない場合がある
- シンボル名だけでは型が分からないため、詳細は `hover` で補完する

---

### definition — 定義元へのジャンプ

指定した位置のシンボル（関数・クラス・変数）の定義元ファイルと行番号を返す。

**使いどころ:**
- 実行パス追跡で「この関数の中身を読みたい」ときに使う
- インポートされた外部シンボルの実装を確認する
- インターフェースの実装クラスを探す

**使い方の流れ:**
1. `Read` または `Grep` でシンボルの使用箇所を特定する
2. `LSP definition` で定義元にジャンプする
3. 定義元を `Read` して実装を読む
4. その定義元でさらに `LSP definition` を繰り返して呼び出しチェーンを下る

**Grep との使い分け:**
- `Grep` はテキストパターンを検索するため、同名関数が複数ある場合に誤検知する
- `LSP definition` は型情報を使って正確な定義元を返すため、オーバーロードや同名関数でも正確

---

### hover — 型情報・ドキュメントの取得

指定した位置のシンボルの型シグネチャ・JSDoc/docstring・インターフェース定義などを返す。

**使いどころ:**
- 関数のパラメータ型・戻り値型を確認する
- インターフェース・型エイリアスの定義内容を素早く確認する
- 外部ライブラリの関数シグネチャを確認する（定義元を読む前に概要を把握）
- ジェネリクス・型パラメータの実際の型を理解する

**使い方の流れ:**
1. 確認したいシンボルのファイルと行・列位置を特定する（`Read` で行番号を確認）
2. `LSP hover` で型情報を取得する
3. 戻り値の型が別の型定義の場合、さらに `LSP definition` でその型定義を確認する

**活用パターン — データフローの追跡:**
```
関数Aの戻り値型を hover で確認
→ その型が関数Bのパラメータとして渡されることを Grep で確認
→ 関数Bの実装を definition でジャンプして読む
→ データ変換がどこで起きるかを理解する
```

---

### findReferences — 参照箇所の全列挙

指定したシンボルが参照されているすべての箇所（ファイル・行番号）を返す。

**使いどころ:**
- 「このクラス・関数がどこで使われているか」を網羅的に調べる
- 影響範囲調査（変更したとき何が壊れるか）
- 層間の通信パターンを確認する（サービス層がどこから呼ばれているか等）
- 依存グラフのマッピング

**使い方の流れ:**
1. 調査対象のシンボルのファイルと行・列位置を特定する
2. `LSP findReferences` で全参照箇所を取得する
3. 参照箇所のファイル群からアーキテクチャ層のパターンを読み取る
4. 多く参照されているシンボルは「中心的な抽象」として重要度が高い

**Grep との使い分け:**
- `Grep` は文字列マッチのため、変数名の一部が一致する別のシンボルも拾う
- `LSP findReferences` はセマンティックな参照のみを返す（同名の別シンボルは含まない）
- 大規模コードベースでは `findReferences` の精度が重要

## 探索シナリオ別の操作パターン

### シナリオ1: 未知の機能を理解する

```
1. Glob でディレクトリ構造を把握する
2. LSP documentSymbols でキーファイルのシンボル一覧を取得する
3. 関連しそうなシンボルを LSP hover で型確認する
4. LSP definition で実装を読む
5. 実装内で呼ばれる関数を LSP definition で追う
6. 全体の実行フローをまとめる
```

### シナリオ2: 依存関係をマッピングする

```
1. Grep でモジュールのインポート箇所を洗い出す
2. LSP findReferences で各シンボルの参照箇所を取得する
3. 参照元のファイル群からレイヤー構造を読み取る
4. LSP hover で型境界（インターフェース）を確認する
5. 依存グラフを文書化する
```

### シナリオ3: 実行パスを追跡する

```
1. Grep または LSP documentSymbols でエントリポイントを特定する
2. LSP hover でエントリポイントのシグネチャを確認する
3. LSP definition で次の呼び出し先の実装を読む
4. 分岐・条件・非同期処理を記録しながら繰り返す
5. LSP findReferences で逆向きの呼び出し元も確認する
6. フローを整理して出力する
```

### シナリオ4: パターンを認識する

```
1. Glob でファイル命名パターンを確認する
2. LSP documentSymbols で複数ファイルのシンボル構造を比較する
3. Grep で繰り返し現れる命名規則・構造パターンを検索する
4. LSP hover でインターフェース・型エイリアスの定義を確認する
5. コードベース全体の設計原則をまとめる
```

## LSP が使えない場合のフォールバック

LSP サーバーが起動していない、または対応言語でない場合:

| LSP 操作 | フォールバック |
|---------|--------------|
| `documentSymbols` | `Grep` でクラス/関数定義パターンを検索 (`class `, `def `, `function `, `export `) |
| `definition` | `Grep` でシンボル名の定義行を検索 (`grep -n "function funcName"`) |
| `hover` | `Read` で定義箇所を直接読む |
| `findReferences` | `Grep -r` でシンボル名を再帰的に検索 |

フォールバックは精度が落ちるため、同名シンボルの混同に注意する。LSP を使える言語では必ず LSP を優先する。

## 対応言語と LSP サーバー

LSP サーバーの設定は **code-explorer プラグインに `lspServers` として同梱済み**。リポジトリごとに `.lsp.json` を作成する必要はない。

必要な作業は各サーバーのバイナリを PATH に通すことだけ:

| 言語 | サーバー | インストール例 |
|------|---------|--------------|
| TypeScript / JavaScript | typescript-language-server | `npm i -g typescript typescript-language-server` |
| Python | pyright-langserver | `npm i -g pyright` |
| PHP | intelephense | `npm i -g intelephense` |
| Go | gopls | `go install golang.org/x/tools/gopls@latest` |
| Java | jdtls | `brew install jdtls` |

これ以外の言語（Ruby・Rust・C++ 等）は LSP が使えないため、Grep/Read でフォールバックする。

### TypeScript / JavaScript の注意点

`typescript-language-server` はワークスペースに `typescript` パッケージ（`node_modules/typescript`）が存在することを要求する。`npm install` 前や `typescript` を依存に持たないリポジトリではサーバーが起動エラーになる。その場合は下記フォールバック表の Grep/Read を使うか、`npm i -g typescript` でグローバルインストールする。
