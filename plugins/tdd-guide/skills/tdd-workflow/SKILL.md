# TDDワークフロースキル

すべてのコード開発が TDD 原則と包括的なテストカバレッジに従って進むよう保証するスキル。

## 有効化のタイミング

- 新しい機能や処理を実装するとき
- バグや問題を修正するとき
- 既存コードをリファクタリングするとき
- API エンドポイントを追加するとき
- 新しいコンポーネントを作成するとき

## コア原則

### 1. テストを先に書く
**常に**テストを先に書き、それをパスさせるために実装コードを書く。

### 2. カバレッジ要件
- ユニット + 統合 + E2E で最低 80% のカバレッジ
- すべてのエッジケースをカバーする
- エラーシナリオをテストする
- 境界条件を検証する

### 3. テストタイプ

**ユニットテスト**
- 個別の関数・ユーティリティ
- コンポーネントロジック
- 純粋関数
- ヘルパーとユーティリティ

**統合テスト**
- API エンドポイント
- データベース操作
- サービス間の相互作用
- 外部 API 呼び出し

**E2E テスト**
- クリティカルなユーザーフロー
- 完全なワークフロー
- ブラウザの自動化
- UI インタラクション

## Git チェックポイント

リポジトリが Git 管理下にある場合、各 TDD ステージの後にチェックポイントコミットを作成する:

- **RED 検証後**: テスト追加のコミット
- **GREEN 検証後**: 最小実装のコミット
- **リファクタ完了後**: リファクタのコミット（任意）

**遵守事項:**
- チェックポイントコミットはワークフロー完了前にスカッシュ・リライトしない
- 各コミットメッセージにはステージと取得した証跡を記述する
- 現在のタスクで現在のアクティブブランチに作成したコミットのみをカウントする
- 他ブランチ・無関係な過去の作業・ブランチ履歴の遠いコミットは有効なチェックポイント証跡と見なさない
- チェックポイントを満たしたと判断する前に、そのコミットが現ブランチの `HEAD` から到達可能であり現タスクのシーケンスに属することを検証する

**推奨コミットメッセージ形式:**
- RED: `test: add reproducer for <feature or bug>`
- GREEN: `fix: <feature or bug>`
- REFACTOR: `refactor: clean up after <feature or bug> implementation`

## TDD ワークフロー手順

### Step 1: ユーザージャーニーを書く
```
[役割] として、[アクション] をしたい。なぜなら [利益] があるから。

例:
ユーザーとして、市場を意味的に検索したい。
なぜなら正確なキーワードがなくても関連する市場を見つけられるから。
```

### Step 2: テストケースを生成する
各ユーザージャーニーに対して、網羅的なテストケースを作成する:
- 正常系（期待する振る舞い）
- エッジケース（null・空・境界値）
- フォールバック動作
- ソート・フィルタリングのロジック

具体的なテストパターンは `references/test-patterns.md` を参照。

### Step 3: テストを実行して失敗を確認する（RED）

テストを実行する（FW に応じて）:
```bash
# JS/TS
npm test

# Python
pytest

# Go
go test ./...
```

**このステップは必須であり、すべての本番変更の RED ゲートとなる。**

ビジネスロジックやその他の本番コードを変更する前に、以下のいずれかのパスで有効な RED 状態を確認する:

- **ランタイム RED**:
  - 対象のテストターゲットが正常にコンパイルされる
  - 新しいまたは変更されたテストが実際に実行される
  - 結果が RED である
- **コンパイル時 RED**:
  - 新しいテストがバグのあるコードパスを新たにインスタンス化・参照・実行する
  - コンパイル失敗そのものが意図した RED シグナルである

いずれの場合も、失敗は意図したビジネスロジックのバグ・未定義の動作・未実装によるものでなければならない。無関係な構文エラー・壊れたテストセットアップ・依存関係の欠落・無関係なリグレッションだけが原因の失敗は対象外。

**書いただけでコンパイル・実行されていないテストは RED と見なさない。**

### Step 4: 実装コードを書く
テストをパスさせるための最小限のコードを書く:

```
// テストによって誘導された実装
// 必要十分な最小限のコードのみ記述する
```

リポジトリが Git 管理下にある場合、最小限の修正をステージングしてから Step 5 の GREEN 検証を待ってコミットする。

### Step 5: テストを再実行して成功を確認する（GREEN）
同じ対象テストターゲットを再実行し、以前失敗していたテストが GREEN になったことを確認する。

有効な GREEN 結果が得られた後にのみリファクタリングに進む。

リポジトリが Git 管理下にある場合、GREEN 検証直後にチェックポイントコミットを作成する。

### Step 6: リファクタリング
テストをグリーンに保ちながらコード品質を向上させる:
- 重複を除去する
- 命名を改善する
- パフォーマンスを最適化する
- 可読性を高める

リポジトリが Git 管理下にある場合、リファクタリング完了後にテストが引き続きグリーンであることを確認してからチェックポイントコミットを作成する。

### Step 7: カバレッジを確認する
カバレッジレポートを実行し、80%以上のカバレッジを確認する。

カバレッジコマンドと閾値の設定方法は `references/framework-guide.md` を参照。

## エッジケースとアンチパターン

### 必ずテストすべきエッジケース
- `null` / `undefined` 入力
- 空の配列・文字列
- 不正な型が渡された場合
- 境界値（最小値・最大値）
- エラーパス（ネットワーク障害・DB エラー）
- 競合状態（並行操作）
- 大量データ（1万件以上）
- 特殊文字（Unicode・絵文字・SQL 特殊文字）

### 避けるべきアンチパターン
- 振る舞いではなく実装の詳細をテストする
- 共有状態によるテスト間の依存
- アサーションが不十分で何もチェックしていない通過テスト
- 外部依存（DB・キャッシュ・外部 API・LLM 等）をモックしない

アンチパターンの具体的な例と対策は `references/test-patterns.md` を参照。

## ベストプラクティス

1. **テストを先に書く** — 常に TDD
2. **1テスト1アサーション** — 単一の振る舞いにフォーカス
3. **説明的なテスト名** — テストされている内容を説明する
4. **Arrange-Act-Assert** — 明確なテスト構造
5. **外部依存をモックする** — ユニットテストを分離する
6. **エッジケースをテストする** — null・undefined・空・大量
7. **エラーパスをテストする** — ハッピーパスだけでない
8. **テストを高速に保つ** — ユニットテストは 1 件 50ms 未満
9. **テスト後にクリーンアップ** — 副作用を残さない
10. **カバレッジレポートを確認する** — ギャップを特定する

## 成功指標

- 80% 以上のコードカバレッジを達成している
- すべてのテストがパスしている（グリーン）
- スキップ・無効化されたテストがない
- テスト実行が高速（ユニットテストは 30 秒未満）
- E2E テストがクリティカルなユーザーフローをカバーしている
- テストが本番前のバグを捕捉している

---

**覚えておくこと**: テストはオプションではない。自信を持ってリファクタリングし、迅速に開発し、本番の信頼性を確保するためのセーフティネットである。
