# レポート・テンプレートリファレンス

セキュリティレビューの出力フォーマット、PRコメントテンプレート、CRITICAL検出時の緊急対応チェックリスト。

---

## 1. セキュリティレビューレポートテンプレート

### 指摘フォーマット

各指摘は以下の形式で記述する:

```
[CRITICAL] ソースコードにAPIキーがハードコードされている
ファイル: src/api/client.ts:42
問題: APIキー "sk-abc..." がソースコードに露出している。
      git の履歴に残り、リポジトリを閲覧できる人物が全員アクセスできる。
修正: 環境変数に移動し、.gitignore と .env.example に追加する。

  // 悪い例
  const apiKey = "sk-abc123";

  // 良い例
  const apiKey = process.env.OPENAI_API_KEY;
  // .env.example に追記: OPENAI_API_KEY=your-api-key-here
```

### 重大度の定義

| 重大度 | 説明 | 対応方針 |
|--------|------|---------|
| CRITICAL | 悪用すると即座に実害につながる問題（認証情報漏洩・インジェクション・認証バイパス） | マージ前に必ず修正。シークレット露出の場合は即時ローテーション。 |
| HIGH | 特定の条件下で悪用可能な問題（SSRF・CSRF・レート制限欠落・競合状態） | マージ前に修正推奨。リスクを理解した上でのマージは可。 |
| MEDIUM | 多層防御を弱める問題（ログへの機密漏洩・エラーメッセージ漏洩・セキュリティヘッダー欠落） | 次のリリースまでに対応する。 |
| LOW | ベストプラクティスからの逸脱（HTTPS未強制・不十分なログ記録） | バックログに追加して計画的に対応する。 |

### サマリーフォーマット

すべてのレビューを以下のサマリーで締めくくる:

```
## セキュリティレビューサマリー

| 重大度   | 件数 | 状態 |
|---------|------|------|
| CRITICAL | 0   | ✅ pass |
| HIGH     | 2   | ⚠️ warn |
| MEDIUM   | 1   | ℹ️ info |
| LOW      | 0   | ✅ pass |

判定: WARNING — マージ前に2件の HIGH 問題を解決することを推奨。
```

### 判定基準

| 判定 | 条件 |
|------|------|
| **APPROVE** | CRITICALもHIGHも0件。クリーンなレビューは有効な結果。 |
| **WARNING** | CRITICALは0件だがHIGHが1件以上。注意してマージ可能。 |
| **BLOCK** | CRITICALが1件以上。マージ前に必ず修正する。 |

## 2. PRレビューコメントテンプレート

### 単一指摘コメント

```
🔴 **[CRITICAL] SQLインジェクション**

**場所**: `src/db/queries.ts:87`

**問題**: ユーザー入力が文字列連結でSQLクエリに埋め込まれており、攻撃者が任意のSQLを実行できます。

```typescript
// 現在のコード（危険）
const query = `SELECT * FROM users WHERE email = '${email}'`;
```

**修正案**:

```typescript
// パラメータ化クエリを使用する
const query = `SELECT * FROM users WHERE email = $1`;
const result = await db.query(query, [email]);
```

マージ前にこの問題を修正することをお願いします。
```

### セキュリティ承認コメント

```
✅ **セキュリティレビュー: APPROVE**

変更を確認しました。セキュリティ上の問題は見つかりませんでした。

- ハードコードされたシークレット: なし
- インジェクション脆弱性: なし
- 認証バイパス: なし
- 依存関係の脆弱性: npm audit クリーン

マージ可能です。
```

### セキュリティ警告コメント

```
⚠️ **セキュリティレビュー: WARNING**

変更を確認しました。以下のHIGH問題が見つかりました。修正を推奨しますが、リスクを理解した上でのマージは許容されます。

**HIGH: レート制限欠落**
`POST /api/auth/login` にレート制限がなく、ブルートフォース攻撃を受けやすい状態です。
`express-rate-limit` の追加を推奨します。

詳細は以下のドキュメントを参照: [express-rate-limit](https://github.com/express-rate-limit/express-rate-limit)
```

## 3. CRITICAL検出時の緊急対応チェックリスト

CRITICALの脆弱性を発見した場合は、以下のチェックリストをユーザーに提示する。

### A. シークレット漏洩が検出された場合

```
🚨 CRITICAL: シークレット漏洩を検出しました

以下のステップで対応してください:

[ ] 1. 即時: 露出したシークレット（APIキー・パスワード・トークン）をローテーション/無効化する
[ ] 2. 即時: .gitignore に .env を追加してコミット
[ ] 3. 即時: git の履歴からシークレットを削除する
         git filter-repo --path .env --invert-paths
         または BFG Repo-Cleaner を使用
[ ] 4. コード修正: process.env.XXX に置き換える
[ ] 5. .env.example にプレースホルダーを追加する
[ ] 6. チームに通知し、フォークやクローンがある場合は警告する
[ ] 7. 悪用の痕跡がないかログを確認する

詳細: OWASP Secrets Management Cheat Sheet
```

### B. インジェクション脆弱性が検出された場合

```
🚨 CRITICAL: インジェクション脆弱性を検出しました

以下のステップで対応してください:

[ ] 1. 即時: 影響を受けるエンドポイントへのアクセスを制限または一時停止を検討する
[ ] 2. コード修正: パラメータ化クエリ / 安全なAPIに置き換える
[ ] 3. 悪用された痕跡がないかログを確認する（異常なクエリ・エラー増加）
[ ] 4. 修正後: テストで修正を確認する
[ ] 5. 修正後: マージしてデプロイする
```

### C. 認証バイパスが検出された場合

```
🚨 CRITICAL: 認証バイパスを検出しました

以下のステップで対応してください:

[ ] 1. 影響を受けるエンドポイントとデータを特定する
[ ] 2. 不正アクセスの痕跡がないかログを確認する
[ ] 3. コード修正: 認証ミドルウェアを追加する
[ ] 4. 修正後: すべての保護ルートをテストする
[ ] 5. 必要に応じてセキュリティチームまたはプロジェクトオーナーに通知する
```

## 4. セキュリティレビュー完了チェックリスト

レビュー完了前に確認する項目:

```
セキュリティレビュー完了チェックリスト

[ ] ハードコードされたシークレットがないことを確認
[ ] SQLインジェクションがないことを確認
[ ] XSS脆弱性がないことを確認
[ ] 認証チェックがすべての保護ルートに存在することを確認
[ ] ユーザー入力のバリデーションが実施されていることを確認
[ ] npm audit でCRITICAL/HIGHの脆弱性がないことを確認
[ ] SSRF対策（URLホワイトリスト等）が実施されていることを確認
[ ] レート制限が認証エンドポイントに設定されていることを確認
[ ] ログに機密情報が記録されていないことを確認
[ ] エラーレスポンスが内部情報を漏洩していないことを確認
```
