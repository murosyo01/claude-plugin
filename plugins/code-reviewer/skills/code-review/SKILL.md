# コードレビュースキル

git diffまたは指定ファイルを読み込み、セキュリティ・品質・パフォーマンス・ベストプラクティスの観点でコードをレビューする。重大度別（CRITICAL / HIGH / MEDIUM / LOW）に指摘を分類し、APPROVE / WARNING / BLOCK の判定を返す。

## レビュープロセス

### 1. コンテキストの収集

`git diff --staged` と `git diff` を実行してすべての変更を確認する。差分がない場合は `git log --oneline -5` で最近のコミットを確認し、ファイルを直接読み込む。引数でファイルが指定された場合はそのファイルを優先する。

### 2. スコープの把握

- 変更されたファイルと関連する機能・修正を特定する
- ファイル全体を読み込み、インポート・依存関係・呼び出し元を把握する
- 変更を単独でレビューせず、周辺コードをコンテキストとして使う

### 3. チェックリストの適用

以下の優先順位でチェックする:

**セキュリティ (CRITICAL)**
- ソースコードにハードコードされた認証情報（APIキー・パスワード・トークン）
- SQLインジェクション：パラメータ化クエリではなく文字列連結
- XSS脆弱性：HTML/JSXでエスケープされていないユーザー入力
- パストラバーサル：サニタイズなしのユーザー制御ファイルパス
- CSRF脆弱性・認証バイパス・脆弱な依存関係
- ログへの機密情報（トークン・パスワード・PII）漏洩

**コード品質 (HIGH)**
- 関数が50行超、ファイルが800行超
- 4段以上の深いネスト
- 未処理のPromise拒否・空のcatchブロック
- ミューテーションパターン（スプレッド・map・filterを優先）
- マージ前のconsole.logステートメント
- テストカバレッジのない新しいコードパス
- コメントアウトされたコード・未使用インポート

**React/Next.js (HIGH)**
- 不完全な依存配列を持つuseEffect/useMemo/useCallback
- リスト要素のキー欠落（インデックスをキーとして使用）
- サーバーコンポーネントでのuseState/useEffect
- フォールバックUIのないデータフェッチング

**Node.js/バックエンド (HIGH)**
- スキーマバリデーションなしで使用するリクエストボディ/パラメータ
- スロットリングのない公開エンドポイント
- LIMITなしのSELECT *
- ループ内での関連データ取得（N+1クエリ）
- タイムアウトのない外部HTTP呼び出し

**パフォーマンス (MEDIUM)**
- O(n^2)なアルゴリズム（O(n)が可能な場合）
- メモ化欠落による不要な再レンダリング
- ライブラリ全体のインポート（ツリーシェイク可能な場合）

**ベストプラクティス (LOW)**
- チケット番号のないTODO/FIXME
- パブリックAPIのドキュメント欠落
- 説明のないマジックナンバー

### 4. 確信度ベースのフィルタリング

指摘を書く前に以下4点を確認する。「いいえ」または「不明」があれば重大度を下げるか削除する:

1. 正確な行を引用できるか（ファイル名と行番号）
2. 具体的な失敗シナリオを説明できるか（入力・状態・悪い結果）
3. 周辺のコンテキスト（呼び出し元・インポート・テスト）を読んだか
4. 重大度は正当か

**確信度80%未満の指摘はスキップする。**

HIGH/CRITICALには以下の証拠が必要:
- 正確なスニペットと行番号
- 具体的な失敗シナリオ
- 既存ガード（型・バリデーション・フレームワーク）がなぜ機能しないか

**スキップすべきよくある誤検知:**
呼び出し元がバリデーション済みの内部関数への「入力バリデーション欠落」・フレームワークが処理する「エラーハンドリング欠落」・意味が明らかな定数への「マジックナンバー」・意図的なfire-and-forgetへの「awaitが欠落」・JavaScriptファイルへの「TypeScriptを使うべき」

### 5. 結果の報告

各指摘を以下の形式で報告する:

```
[重大度] 問題の概要
ファイル: path/to/file.ts:行番号
問題: 具体的な問題の説明
修正: 推奨される修正方法

  // 悪い例
  // 良い例
```

すべてのレビューを以下のサマリーで締めくくる:

```
## レビューサマリー

| 重大度   | 件数 | 状態 |
|---------|------|------|
| CRITICAL | 0   | pass |
| HIGH     | 0   | pass |
| MEDIUM   | 0   | pass |
| LOW      | 0   | pass |

判定: APPROVE — コードは基準を満たしています。
```

**判定基準:**
- **APPROVE**: CRITICALまたはHIGHの問題なし（指摘ゼロも有効な結果）
- **WARNING**: HIGHの問題のみ（注意してマージ可能）
- **BLOCK**: CRITICALの問題あり（マージ前に必ず修正）

承認を保留して厳密に見えようとしない。差分がクリーンであれば承認する。

## プロジェクト固有のガイドラインの確認

利用可能な場合は `CLAUDE.md` からプロジェクト固有の規約を確認してレビューに反映する:
- ファイルサイズ制限・絵文字ポリシー・イミュータビリティ要件
- データベースポリシー（RLS・マイグレーションパターン）
- エラーハンドリングパターン・状態管理の規約

プロジェクトの確立されたパターンに合わせてレビューを適応させる。
