# コードアーキテクチャスキル

既存コードベースの深い理解に基づき、新機能や変更の実装ブループリストを設計するためのスキル。アーキテクチャの原則論ではなく、「このリポジトリに今日これを実装するには、何をどう作るか」という具体的な実行計画を提供する。

## プロセス

### 1. パターン分析

Read・Grep・Glob・Bash を使ってコードベースを調査する:

- エントリポイント・ディレクトリ構造・モジュール境界を確認する
- 命名規則（ファイル名・クラス名・関数名）のパターンを特定する
- 既に採用されているアーキテクチャパターン（Repository・Service層・ミドルウェア等）を把握する
- テストの配置方法・命名・スタイルを確認する
- 依存グラフを理解してから新しい抽象化を提案する

### 2. アーキテクチャ設計

分析結果をもとに設計方針を決める:

- 既存パターンに自然に馴染む設計を選ぶ（コードベースに統一感を保つ）
- 要件を満たす最もシンプルなアーキテクチャを採用する
- リポジトリがまだ使っていない抽象化を投機的に導入しない
- 変更範囲を明確にし、修正の波及範囲を把握する

### 3. 実装ブループリントの作成

重要なコンポーネントそれぞれについて以下を整理する:

- **ファイルパス** — 作成または変更するファイルの相対パス
- **目的** — 単一の責務
- **主要インターフェース** — エクスポートする型・関数・クラスのシグネチャ
- **依存関係** — 他モジュールや外部ライブラリへの依存
- **データフロー上の役割** — データの入力源と出力先

ブループリントの出力形式については `references/blueprint-template.md` を読み込んで使用する。

### 4. ビルドシーケンスの提示

依存関係に基づいて実装順序を決める:

1. 型・インターフェース定義
2. コアロジック
3. 連携層（API・DB・外部サービス）
4. UI
5. テスト
6. ドキュメント

## 設計原則

### 既存パターンへの適合
コードベースが採用しているパターン・命名規則・ファイル構成を尊重する。新機能は既存の隣接コードを見てその隣に自然に置けるものを設計する。

### シンプルさの優先
投機的な最適化や過剰な抽象化は避ける。「このリポジトリで既に実証されているパターン」を使い、要件を満たす最もシンプルな解を選ぶ。

### 依存関係の明確化
新しい抽象化を提案する前に、既存の依存グラフを理解する。循環依存・密結合の導入を避ける。

### 変更範囲の最小化
作成するファイル・変更するファイルを明確に分け、影響範囲を可能な限り小さく保つ。

## 参照リソース

### 参照ファイル

- **`references/blueprint-template.md`** — ブループリントの出力テンプレートとビルドシーケンスの記入指針。ブループリストを出力する前に読み込む。
