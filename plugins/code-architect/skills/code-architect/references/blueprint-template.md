# ブループリント出力テンプレート

ブループリストを出力するときはこのテンプレートを使う。見出しの順序と形式を維持し、各セクションを漏れなく埋める。

---

## テンプレート

```markdown
## アーキテクチャ: [機能名]

### 設計上の決定事項
- 決定1: [根拠]
- 決定2: [根拠]

### 作成するファイル
| ファイル | 目的 | 優先度 |
|--------|------|--------|
| path/to/new-file.ts | [単一責務の説明] | 高 / 中 / 低 |

### 変更するファイル
| ファイル | 変更内容 | 優先度 |
|--------|---------|--------|
| path/to/existing.ts | [変更内容の要点] | 高 / 中 / 低 |

### データフロー
[リクエスト→処理→レスポンスの流れ、またはイベント駆動の場合はイベント発行から購読までの流れを記述する]

### ビルドシーケンス
1. [ステップ1（型・インターフェース定義）]
2. [ステップ2（コアロジック）]
3. [ステップ3（連携層）]
4. [ステップ4（UI）]
5. [ステップ5（テスト）]
6. [ステップ6（ドキュメント）]
```

---

## 各セクションの記入指針

### 設計上の決定事項
取り消しが難しい選択・複数ファイルに影響する設計判断のみ記録する。例:

- 「サービス層を新設する: 既存の routes/ はすべて薄いオーケストレーターとなっているため」
- 「イベント駆動でなく同期処理を選択: 通知量が少なくキューインフラのコストに見合わないため」

### 作成するファイル
優先度の目安:
- **高**: 他のファイルがこれに依存する（型定義・インターフェース・コアサービス）
- **中**: 機能を完成させるが直接の依存は少ない（連携層・ハンドラ）
- **低**: なくても動くが品質に必要（テスト・ドキュメント）

### 変更するファイル
変更内容は「何を追加/削除/修正するか」を具体的に書く。例:

- 「新しいサービスをDIコンテナに登録する」
- 「ルートに `/notifications` エンドポイントを追加する」

### データフロー
フロー記述のパターン:

**HTTPリクエスト型:**
```
クライアント → ルートハンドラ → バリデーション → サービス層 → リポジトリ → DB
                                                ↓
                                          レスポンス ← DTO変換
```

**イベント型:**
```
ドメインイベント発行 → イベントバス → 購読者A（メール送信）
                                   → 購読者B（ログ記録）
```

**バッチ型:**
```
スケジューラ → ジョブ → データ取得 → 変換処理 → 書き込み → 完了通知
```

### ビルドシーケンス
依存関係に沿った順序で書く。各ステップに「なぜその順か」を1行添えると実装者が判断しやすい。

例:
```
1. src/types/notification.ts — 他のすべてがこの型に依存するため最初に作る
2. src/services/NotificationService.ts — コアロジックをDIなしでテスト可能にする
3. src/repositories/NotificationRepository.ts — DBアクセスをサービスから切り離す
4. src/routes/notifications.ts — HTTPレイヤーを最後に薄くつなぐ
5. src/services/__tests__/NotificationService.test.ts — ユニットテストをコアロジックに集中させる
6. docs/notifications.md — 実装後に実際の挙動を文書化する
```

---

## 注意事項

- ファイルパスはリポジトリ root からの相対パスで書く
- 既存ファイルを変更する場合は「変更するファイル」に記載し、「作成するファイル」には含めない
- 優先度は依存関係をもとに決める（被依存ファイルほど優先度が高い）
- ビルドシーケンスは「作成するファイル」と「変更するファイル」を合わせた順序で提示する
