# アーキテクチャパターン参照

よく使われるパターンの詳細な説明・ユースケース・使い分けの指針。

---

## フロントエンドパターン

### コンポーネント合成
1つのことに集中したシンプルなコンポーネントから複雑なUIを構築する。

**使うべき場面**: 常に。UIの標準アプローチ。
**トレードオフ**: ファイル数は増えるが、各コンポーネントがシンプルでテストしやすくなる。
**例**: `Avatar`・`UserName`・`Badge` コンポーネントを組み合わせた `UserCard`。

### Container/Presenter（Smart/Dumb）
データを取得するコンポーネント（Container）とデータをレンダリングするコンポーネント（Presenter）を分離する。

**使うべき場面**: UIコンポーネントがAPIやグローバル状態からデータを必要とする場合。
**トレードオフ**: コンポーネント数は増えるが、Presenterは純粋な関数になりテスト・再利用が容易になる。
**避けるべき場面**: 分割するメリットがないほどシンプルなコンポーネントの場合。

### カスタムフック
ステート付きのロジックを再利用可能なフックに抽出する。

**使うべき場面**: 同じステート付きロジック（データ取得・フォーム処理・サブスクリプション）が複数のコンポーネントに現れる場合。
**例**: `useDebounce`・`usePagination`・`useWebSocket`。

### グローバル状態にContext
React Context（または同等のもの）を使ってプロップのバケツリレーなしに状態を共有する。

**使うべき場面**: 本当のグローバル状態 — 認証ユーザー・テーマ・ロケール。頻繁に変わるデータには使わない（不必要な再レンダリングを引き起こす）。
**推奨**: 多くの更新を伴う複雑な状態には状態管理ライブラリ（Zustand・Redux）を使用する。

### コード分割
ルートと重いコンポーネントを遅延ロードして初期バンドルサイズを削減する。

**使うべき場面**: 初回ロード時に不要なルートまたは30KB超のコンポーネント。
**実装**: `React.lazy` + `Suspense`、またはフレームワーク組み込み（`next/dynamic`）。

## バックエンドパターン

### Repositoryパターン
データアクセスをインターフェースの背後に隠蔽し、ビジネスロジックがデータベースに依存しないようにする。

**使うべき場面**: ストレージ実装の差し替えや、実DBなしでビジネスロジックを単体テストしたい場合。
**構造**:
```
UserRepository（インターフェース）
  ├── PostgresUserRepository
  └── InMemoryUserRepository（テスト用）
```

### サービス層
ビジネスロジックをHTTPハンドラ・コントローラから切り離す。

**使うべき場面**: 非trivialなアプリケーションでは常に。ハンドラは薄いオーケストレーターになる。
**利点**: HTTPサーバーを起動せずにビジネスロジックをテストできる。

### ミドルウェアパイプライン
リクエスト・レスポンス処理を小さな単一責任の関数として合成する。

**使うべき場面**: 横断的関心事 — 認証・ログ・レート制限・エラーフォーマット。
**例**: `auth → rateLimit → validate → handler → errorHandler`

### イベント駆動アーキテクチャ
イベントバスまたはメッセージキューを使って生産者と消費者を疎結合にする。

**使うべき場面**:
- レスポンス前に完了する必要のない副作用（メール・アナリティクス）
- 独立して進化すべきサービスの疎結合化
- 負荷時にバッファリングが必要なワークロード

**トレードオフ**: 運用上の複雑さが増す（キューインフラ・デッドレターキュー・冪等性）。

### CQRS（コマンドクエリ責務分離）
読み取りモデルと書き込みモデルを分離する。

**使うべき場面**: 読み取りパターンと書き込みパターンが大きく異なるドメイン（例: ダッシュボードが集計データをクエリする一方で原子的に書き込む）。
**避けるべき場面**: シンプルなCRUDアプリケーションへの過剰設計。

## データパターン

### 正規化スキーマ
各データを1箇所のみに保存し、外部キーでエンティティを関連付ける。

**使うべき場面**: 書き込みが多いワークロード・データ整合性が重要な場合・関係が複雑な場合。
**トレードオフ**: クエリ時にJOINが必要。インデックスを丁寧に設計しないと規模によっては遅くなる。

### 読み取り用非正規化
クエリ時の高コストなJOINを避けるためにデータを複製または事前集計する。

**使うべき場面**: 読み取りが多いワークロード・ダッシュボード・検索インデックス・レコメンドシステム。
**トレードオフ**: 書き込み時に複数箇所を更新する必要がある。同期しない場合は古いデータが読まれるリスク。

### イベントソーシング
現在の状態ではなく不変のイベントログとして状態を保存する。

**使うべき場面**: 監査証跡の要件・時制クエリを含む複雑なドメイン・再生が必要なシステム。
**避けるべき場面**: シンプルなCRUDアプリケーション — 大幅に複雑さが増す。

### キャッシュ戦略

| 層 | ツール | TTL | ユースケース |
|---|---|---|---|
| インプロセス | LRUキャッシュ | 秒単位 | ホットな設定・ルックアップテーブル |
| 分散 | Redis | 分〜時間 | セッションデータ・計算済み結果 |
| CDN | CloudFront / Fastly | 時間〜日 | 静的アセット・パブリックAPIレスポンス |
| DB | クエリキャッシュ | 秒単位 | 頻繁に実行される同一クエリ |

**無効化アプローチ**:
- **TTLベース**: シンプルだが古いデータを返す可能性がある
- **書き込み時更新**: 書き込み時に無効化して一貫性を保つが、書き込みレイテンシが増加する
- **イベント駆動**: キャッシュ無効化イベントを発行して疎結合にする

### 結果整合性
書き込み後、読み取りが一時的に古いデータを返すことを許容する。

**使うべき場面**: 強い整合性がコスト的に高すぎる分散システム（クロスリージョンレプリケーション・マイクロサービス）。
**前提条件**: 冪等な操作・競合解決戦略・短い遅延へのユーザー許容。

## 統合パターン

### Strangler Fig（ストラングラーフィグ）
新しい機能を新システムにルーティングしながら、既存機能は旧システムで維持してレガシーシステムを段階的に置き換える。

**使うべき場面**: 全面的な書き直しがリスクが高すぎる大規模レガシー移行。

### 腐敗防止層（Anti-Corruption Layer）
2つのシステムのモデルを境界で変換し、互いのモデルが相手に漏れないようにする。

**使うべき場面**: サードパーティAPIやレガシーシステムと統合する際に、そのデータモデルがドメインモデルに侵入しないようにする。

### 指数バックオフ付きリトライ
増加する遅延とジッターで一時的な失敗をリトライする。

**使うべき場面**: すべての外向きHTTPコール・DBコネクション・メッセージコンシューマー。
**式**: `delay = min(base * 2^attempt + jitter, max_delay)`

### サーキットブレーカー
失敗している依存先への呼び出しを一時停止して回復時間を与え、カスケード障害を防ぐ。

**使うべき場面**: 外部サービスへのすべての呼び出し。
**状態**: Closed（正常）→ Open（失敗中・呼び出し拒否）→ Half-open（回復を確認中）
