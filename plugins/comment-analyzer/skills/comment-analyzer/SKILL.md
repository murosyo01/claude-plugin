# コメント分析スキル

指定されたファイルのコメントを読み込み、**まずフロントエンドかどうかを判定する**。フロントエンドファイルはブラウザの検証ツールでコメントが閲覧されるため実際に削除する。非フロントエンドファイルはコードと突合して Inaccurate / Stale / Incomplete / Low-value の4軸でアドバイザリ分析を行う。

詳細なパターン・良い例/悪い例は `references/patterns.md` を参照すること。

## 分析プロセス

### 1. 対象の特定

引数で指定されたファイルを `Read` で読み込む。関連ファイル・依存関係の確認が必要な場合は `Grep` / `Glob` を使う。対象が指定されていない場合はユーザーに確認する。

### 2. フロントエンド判定

以下の拡張子はフロントエンドファイルと判定し、**手順 2a（コメント削除）** に進む。それ以外は **手順 3（コードとの突合）** に進む。

フロントエンド拡張子: `.html` / `.htm` / `.jsx` / `.tsx` / `.vue` / `.svelte` / `.css` / `.scss` / `.less` / `.styl`

`.js` / `.ts` は `document`・`window`・`localStorage`・`navigator`・`addEventListener` などブラウザグローバルへの参照があればフロントエンドと判定する。

### 2a. フロントエンドコメントの削除（フロントエンドのみ）

1. ファイル内のすべてのコメントを行番号付きで列挙する
2. 保持するコメントを除外する:
   - `/*!` で始まるライセンス保持コメント
   - `/* webpackChunkName: */`・`/* @vite-ignore */`・`/* #__PURE__ */` などビルドツール指示コメント
3. 残りを `Edit` で削除する。インラインコメントはコード行を残しコメント部分のみ削除する。JSX の `{/* ... */}` は `{}` ごと削除する
4. 削除後に空行が過剰に残る場合はまとめて除去する
5. 削除したコメント数とファイル名を報告して**終了する**

削除対象の構文:

| 種別 | 構文 |
|------|------|
| HTML / Vue テンプレート / Svelte | `<!-- ... -->` |
| JS / TS の行コメント | `// ...` |
| JS / TS のブロックコメント | `/* ... */` |
| JSX / TSX の式コメント | `{/* ... */}` |
| CSS / SCSS / LESS / Styl | `/* ... */` |

### 3. コードとコメントの突合

- コメントとそれが説明している実装コードを並べて確認する
- 引数・戻り値・ロジック・副作用の説明が実装と一致しているかを検証する
- 実装全体の文脈を読み、変更後に更新されなかったコメントを特定する

### 3. 4軸チェックの適用

**Inaccurate（不正確）**
- コメントの説明が実装と矛盾している
- 引数・戻り値の型・単位・意味が食い違っている
- 実際のコードと矛盾する動作を記述している

**Stale（古い）**
- 削除・変更された挙動への参照が残っている
- リファクタリング後に更新されていない引数名・型・戻り値の説明
- 存在しないモジュール・関数・クラスへの参照

**Incomplete（不完全）**
- 複雑なアルゴリズムやビジネスルールに説明がない
- 重要な副作用・前提条件・制約が記述されていない
- パブリックAPIで引数・戻り値・スロー例外の記述が欠落している

**Low-value（価値が低い）**
- 実装をそのまま言い換えているだけのコメント
- コードを読めば自明な説明（例: `// カウントをインクリメントする`）
- 将来の読者に何も付加価値を与えない記述

### 4. 確信度フィルタ

指摘を書く前に以下を確認する:

1. 正確な行を引用できるか（ファイル名と行番号）
2. コードと突合して矛盾を確認したか（推測でフラグしない）
3. 分類は正当か（スタイルの好みをInaccurateに分類しない）

**確信度80%未満の指摘はスキップする。**

スキップすべき誤検知: 名前から自明な内部ヘルパーへの「コメント不足」・ライセンスヘッダーや生成コードなど慣習的な定型コメント・プロジェクト全体の言語規約に合ったコメント形式・単純な1目的関数への「説明が短すぎる」

### 5. 分類して報告

各指摘を以下の形式で出力する:

```
[タグ] 指摘の概要
ファイル: path/to/file.ts:行番号
問題: 具体的な問題の説明
提案: 推奨される修正方向
```

すべての分析を以下のサマリーで締めくくる:

```
## 分析サマリー

| タグ         | 件数 |
|-------------|------|
| Inaccurate  | 0    |
| Stale       | 0    |
| Incomplete  | 0    |
| Low-value   | 0    |

総指摘数: 0 件 — コメントは現在の実装と一致しており、品質上の問題は見当たりません。
```

指摘がゼロのサマリーは有効かつ期待される結果である。問題を作り出してサマリーを埋めない。
