# コード簡潔化スキル

最近変更されたコードまたは指定ファイルを読み込み、振る舞いを完全に保ったまま構造・可読性・品質の観点で簡潔化する。

## 実行手順

### 1. 対象の特定

引数でファイルが指定された場合はそのファイルを優先する。指定がない場合は以下を実行する：

```bash
git diff --staged   # ステージング済みの変更
git diff            # 未ステージの変更
```

差分がない場合は `git log --oneline -5` で最近のコミットを確認し、直近のファイルを対象とする。

### 2. 周辺コードの読み込み

変更を単独で見ない。対象ファイルを全体的に読み込み、以下を把握する：
- インポートと依存関係
- 呼び出し元（Grep で参照元を確認）
- プロジェクトの命名規則・スタイル

### 3. 簡潔化候補の抽出

以下のカテゴリで明確に改善できる箇所を特定する：

**構造**
- 深いネスト（3段以上）→ 名前付き関数への抽出 または 早期リターン
- コールバックチェーン → `async` / `await` への変換
- デッドコード・未使用インポート → 除去

**可読性**
- 意味が不明な短い変数名 → 記述的な名前に変更
- ネストした三項演算子 → 条件分岐または中間変数に分解
- アクセスが不明確なオブジェクト参照 → 分割代入の活用

**品質**
- デバッグ用 `console.log` → 除去
- コメントアウトされたコード → 除去
- 重複したロジック → 統合
- 一箇所からしか呼ばれない単一目的ヘルパー → インライン化

### 4. 機能的に等価な変更のみ適用

**振る舞いを変える変更は行わない。** 不確かな場合は変更しない。

以下はスコープ外：
- バグ修正・機能追加
- 変更されていないコードへの手入れ（明示指示がない限り）
- プロジェクトの慣習に反するスタイル変更
- 将来の使用を想定した新しい抽象化の導入

### 5. 検証

変更後に以下を確認する：
- テストが存在する場合は実行する（`npm test` / `pytest` / 相当するコマンド）
- 変更前後で公開 API・関数シグネチャが変わっていないことを確認する
- 型エラーや構文エラーがないことを確認する（TypeScriptの場合は `tsc --noEmit`）

## 原則

1. **明快さ > 巧妙さ** — 賢いコードより読みやすいコードを優先する
2. **既存スタイルとの一貫性** — プロジェクトの慣習に合わせる
3. **振る舞いの完全保持** — 機能的に等価な変更のみ
4. **明確に保守性が向上する箇所のみ** — 不確かな改善のために変更を加えない

## 出力フォーマット

変更を適用した後、以下のサマリーを出力する：

```
## 簡潔化サマリー

変更ファイル: path/to/file.ts

### 適用した変更
- [構造] 深いネストを早期リターンで解消（processUsers 関数）
- [品質] デバッグ用 console.log を 3 件除去
- [可読性] ネストした三項演算子を条件分岐に置換

### 変更しなかった理由（あれば）
- getUserById: 呼び出し元が複数あり、インライン化すると重複が増えるため保留
```

変更がなかった場合は「簡潔化できる箇所はありませんでした。コードはすでに明確で保守しやすい状態です。」と報告する。
